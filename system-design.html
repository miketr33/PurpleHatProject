PurpleHat Music Player

System Design Document — .NET 10 / Blazor Server / AWS

1 High-Level Architecture

The application is a Blazor Server web app running on an AWS EC2 instance. Audio files are bundled with the application and served directly by Kestrel — no CDN or S3 involved. Two databases provide persistence: SQLite for relational user data and DynamoDB for key-value session state. A scoped in-memory service preserves playback position during same-tab navigation.

2 AWS Services

The application uses a minimal AWS footprint — just two services:

EC2 — Compute

Single instance with an Elastic IP running the Blazor Server app as a systemd service. The app binary, static assets, and audio files are all deployed here via rsync over SSH from GitHub Actions.

SQLite database file (purplehat.db) lives on the instance's local disk. Audio MP3 files are served directly from wwwroot/audio/ by Kestrel — no S3 or CloudFront needed.

DynamoDB — Session Store

Single table (PlaybackSessions) with on-demand billing (PAY_PER_REQUEST). Stores per-user playback state so users can resume where they left off across browser sessions.

Local dev: DynamoDB Local via Docker on localhost:8000.
Production: AWS DynamoDB service, authenticated via the EC2 instance's default credential chain (IAM role).

Services NOT used (and why)

  • S3 / CloudFront — Audio files are small and bundled; no CDN needed for a demo.
  • RDS — SQLite is sufficient for a single-instance app with seeded users.
  • ALB / API Gateway — Single EC2 instance; no load balancing required.
  • Cognito — Simple demo login with seeded users (no passwords / no auth).

3 Data Persistence — Three Layers

The application uses a three-tier persistence strategy, each layer optimised for its data characteristics: relational user data in SQLite, key-value session state in DynamoDB, and ephemeral playback state in scoped memory.

Layer 1 — Relational
SQLite (EF Core)
Users • Favourites • Health Checks
Survives: everything
Layer 2 — Key-Value
DynamoDB
Playback sessions (track, position, volume)
Survives: browser refresh, new tabs, return visits
Layer 3 — Ephemeral
Scoped In-Memory
Current playback state within a circuit
Survives: page navigation (same tab only)

3a. SQLite — Relational User Data

File: purplehat.db, managed by EF Core with the Microsoft.EntityFrameworkCore.Sqlite provider. Migrations run automatically on startup via db.Database.MigrateAsync().

TablePurposeKeyColumns
Users Seeded demo users Id (int, PK) Id, Name
FavouriteTracks User's favourited tracks Composite: UserId + AudioUrl UserId, AudioUrl
HealthCheckEntries Diagnostic read/write test Id (int, PK) Id, CreatedAt, Message

Key design decision: Favourites are keyed by AudioUrl (filename) rather than a runtime track ID. This means favourites remain stable if the track list order changes or new files are added.

Access pattern: Uses IDbContextFactory<ApplicationDbContext> to create short-lived DbContext instances per operation — the recommended pattern for Blazor Server, which has long-lived scoped services that outlive a typical HTTP request.

Service: TrackFavouriteService Scoped

// In-memory HashSet for instant lookups
private readonly HashSet<string> _favourites = [];

// Toggle: update HashSet immediately, then persist to SQLite if logged in
public async Task ToggleAsync(string audioUrl)
{
    if (_favourites.Contains(audioUrl))
    {
        _favourites.Remove(audioUrl);
        if (userSession.CurrentUser is not null)
            await RemoveFromDbAsync(userSession.CurrentUser.Id, audioUrl);
    }
    else
    {
        _favourites.Add(audioUrl);
        if (userSession.CurrentUser is not null)
            await AddToDbAsync(userSession.CurrentUser.Id, audioUrl);
    }
    OnChange?.Invoke();
}

3b. DynamoDB — Playback Session Persistence

Single table with a simple schema. No sort keys, no GSIs — just one item per user. Table is auto-created on first access via EnsureTableExists().

AttributeTypeRoleDescription
UserId String Partition Key Maps to the seeded user's integer ID
AudioUrl String Attribute The track's file path (e.g. audio/Artist - Title.mp3)
Position Number Attribute Playhead position in seconds (2 decimal places)
Volume Number Attribute Volume level 0–100

Service: PlaybackSessionService Singleton

// Save: PutItem overwrites the user's session record
public async Task SaveAsync(int userId, string audioUrl, double position, int volume)
{
    await EnsureTableExists();
    await dynamoDb.PutItemAsync(new PutItemRequest
    {
        TableName = "PlaybackSessions",
        Item = new Dictionary<string, AttributeValue>
        {
            ["UserId"]   = new(userId.ToString()),
            ["AudioUrl"] = new(audioUrl),
            ["Position"] = new() { N = position.ToString("F2") },
            ["Volume"]   = new() { N = volume.ToString() }
        }
    });
}

// Load: GetItem by UserId partition key
public async Task<PlaybackSession?> LoadAsync(int userId)
{
    await EnsureTableExists();
    var response = await dynamoDb.GetItemAsync(new GetItemRequest
    {
        TableName = "PlaybackSessions",
        Key = new Dictionary<string, AttributeValue>
        {
            ["UserId"] = new(userId.ToString())
        }
    });
    if (!response.IsItemSet) return null;
    // ... map to PlaybackSession record
}

Local vs Production Configuration

// Program.cs — DynamoDB client setup
var dynamoDbServiceUrl = builder.Configuration["DynamoDb:ServiceUrl"];
if (!string.IsNullOrEmpty(dynamoDbServiceUrl))

}
else
{
    // Production: AWS DynamoDB via default credential chain (EC2 IAM role)
    builder.Services.AddSingleton<IAmazonDynamoDB>(new AmazonDynamoDBClient());
}

3c. In-Memory — Playhead During Navigation

PlaybackStateService is registered as Scoped, meaning one instance per Blazor Server circuit (essentially per browser tab). It holds the playback state in plain C# properties — no serialisation, no I/O, instant save and restore.

Service: PlaybackStateService Scoped

public class PlaybackStateService : IPlaybackStateService
{
    public string? AudioUrl { get; private set; }
    public double Position { get; private set; }
    public int Volume { get; private set; } = 60;
    public bool WasPlaying { get; private set; }
    public bool HasState => AudioUrl is not null;

    public void Save(string audioUrl, double position, int volume, bool wasPlaying)
    {
        AudioUrl = audioUrl;
        Position = position;
        Volume = volume;
        WasPlaying = wasPlaying;
    }
}

Why this exists: In Blazor Server, navigating between pages disposes the current page component but keeps the circuit (and its scoped services) alive. Without this service, every page navigation would lose track of what was playing. The in-memory layer provides zero-latency restore — no round-trip to DynamoDB needed for simple navigation.

4 State Lifecycle — How Playback Is Preserved

The three persistence layers work together in a tiered fallback pattern. The fastest layer is tried first; slower layers are only hit when the faster one has no data.

User hits play │ └─▶ JS Audio API begins playing via audioPlayer.js Browser Audio object holds the actual playback state User pauses │ └─▶ SaveStateAsync() fires │ ├─▶ PlaybackStateService.Save() ← in-memory, instant, always │ └─▶ PlaybackSessionService.SaveAsync() ← DynamoDB, async, only if logged in User navigates away from /music │ └─▶ DisposeAsync() fires │ └─▶ SaveStateAsync() ← saves to both layers before component dies User navigates back to /music (same tab) │ └─▶ OnAfterRenderAsync(firstRender: true) │ ├─ PlaybackState.HasState? │ │ │ ├─ YES ──▶ RestoreFromStateAsync() │ │ Fast path: in-memory, no I/O, instant │ │ Restores track, position, volume, and play/pause state │ │ │ └─ NO ──▶ RestoreFromDynamoAsync() │ Slow path: reads from DynamoDB, only if logged in │ Loads track paused at saved position User refreshes the page / opens new tab │ └─▶ New Blazor circuit = new scoped services = in-memory is empty │ └─▶ Falls through to DynamoDB on next visit to /music (if logged in) User logs in │ └─▶ OnUserChanged() fires │ ├─▶ TrackFavourites.LoadForUserAsync() ← loads favourites from SQLite └─▶ RestoreFromDynamoAsync() ← loads playback state from DynamoDB

5 Service Architecture & Lifetimes

All services are registered in Program.cs. The lifetime of each service is chosen based on its data characteristics and thread-safety requirements.

ServiceLifetimePurposeWhy This Lifetime
ITrackService Singleton Loads MP3 files from disk, parses filenames Track list is static; loaded once at startup
IAmazonDynamoDB Singleton AWS DynamoDB client Thread-safe, connection pooling built-in
IPlaybackSessionService Singleton DynamoDB read/write operations Stateless — all state lives in DynamoDB
IUserSessionService Scoped Tracks which user is "logged in" Per-circuit; each tab has its own user
ITrackFavouriteService Scoped In-memory favourites + SQLite sync Per-circuit; each tab has its own favourites set
IPlaybackStateService Scoped In-memory playhead state Per-circuit; navigation state for this tab only
ApplicationDbContext Factory EF Core context for SQLite Short-lived instances avoid long DbContext lifetimes in Blazor

6 CI/CD Pipeline

CI Pipeline (on push + PR to main/master) │ ├─ Checkout ├─ Setup .NET 10 ├─ dotnet restore ├─ dotnet build --configuration Release └─ dotnet test CD Pipeline (on push to main/master only) │ ├─ Checkout ├─ Setup .NET 10 ├─ dotnet publish -c Release -o publish ├─ Setup SSH key from secrets.EC2_SSH_KEY ├─ rsync -avz --delete publish/ → EC2:/home/ec2-user/app/ └─ ssh → sudo systemctl restart purplehat

The deployment is a simple binary copy — no container registry, no blue/green, no rolling deploy. The --delete flag on rsync ensures old files are cleaned up. The systemd service restarts with the new binary automatically.

7 Audio Playback Architecture

Audio playback runs entirely in the browser via the HTML5 Audio API. The Blazor Server component communicates with a JavaScript module (audioPlayer.js) through JS Interop.

Blazor Server (C#) Browser (JS) │ │ │ play(url, volume) ──────────────▶ │ audio.src = url │ │ audio.volume = vol/100 │ pause() ──────────────────────▶ │ audio.pause() │ resume() ──────────────────────▶ │ audio.play() │ setVolume(vol) ────────────────▶ │ audio.volume = vol/100 │ setCurrentTime(t) ─────────────▶ │ audio.currentTime = t │ getCurrentTime() ◀────────────── │ return audio.currentTime │ │ │ OnTrackEnded() ◀─────────────── │ audio 'ended' event │ (auto-advances to next track) │

Why JS Interop? Blazor Server runs on the server with a SignalR connection to the browser. The HTML5 Audio object must live in the browser — it can't be created or controlled from server-side C#. The JS module wraps the Audio API and exposes it as callable functions over interop.

class User
{

Seeded with 3 users: Mike (1), Jordan (2), Jason (3). No passwords — demo-only login.

class FavouriteTrack
{
    int UserId { get; set; }
    string AudioUrl { get; set; }
}
// Composite PK: (UserId, AudioUrl)

Links a user to a track via filename. Composite key prevents duplicates.

PlaybackSession record (DynamoDB)

record PlaybackSession(
    string AudioUrl,
    double Position,
    int Volume
);

C# representation of a DynamoDB item. Keyed by UserId in the table, but the record itself only holds the value attributes.
